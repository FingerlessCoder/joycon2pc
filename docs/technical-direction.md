# 技术方向决策（XInput 模拟 vs 原生驱动）

## 背景
当前目标是“像普通手柄一样稳定玩 PC 游戏”。
现状问题集中在：抖动、误触、偶发延迟、扩展输入（SL/SR、IMU、更多震动能力）不足。

## 两条路线

### 路线 A：继续 XInput（ViGEm）
优点：
- 开发速度快，用户态调试成本低。
- 绝大多数 Windows 游戏直接可用。
- 与现有代码兼容，迁移成本最小。

缺点：
- XInput 能力上限固定，不适合完整表达 Joy-Con 2 全能力。
- IMU、特殊按键、高级震动等需要“折中映射”，不是原生体验。

适用目标：
- 先交付可玩、稳定、延迟可接受版本。

### 路线 B：原生 Windows 驱动 / 自定义 HID
优点：
- 功能上限高，可定义更完整报告与能力集。
- 理论上可更接近 Joy-Con 2 原生特性。

缺点：
- 复杂度高：驱动开发、签名、安装、兼容性、维护都重。
- 迭代慢，风险大，不适合当前“无蓝牙硬件”环境快速验证。

适用目标：
- 长期研发方向，不适合作为当前主线交付。

## 决策建议
短中期采用“双轨策略”：
- 主线：路线 A（ViGEm）持续交付稳定可玩版本。
- 研究线：路线 B 做技术可行性验证，不阻塞主线。

## 分层建议（KISS / DRY / SOLID）
为避免后续重写，统一按 5 层结构推进：
1. Transport：BLE 收发（只处理连接与字节流）
2. Decode：协议解析（字节 -> 结构化状态）
3. Filter：去抖、死区、平滑、节流
4. Mapping：Joy-Con 状态 -> 目标控制器语义
5. Output：ViGEm / 原生 HID / 未来驱动

这能保证以后切换输出层时，不影响前 4 层。

## 无蓝牙硬件的开发策略
当前设备没有蓝牙，建议使用“抓包回放”闭环：
1. 在有蓝牙机器抓原始报告（保存为二进制日志）。
2. 在开发机回放日志，驱动 Decode/Filter/Mapping 全流程。
3. 用同一批日志对比修改前后结果（按键误触率、摇杆漂移、输入延迟代理指标）。

## 验收指标（主线）
- 误触：静止 60 秒内，不出现非预期按键边沿。
- 抖动：摇杆静止输出绝对值长期接近 0（低于死区阈值）。
- 延迟：连接后稳定切到连续上报模式，主观延迟显著下降。
- 兼容：主流 XInput 游戏可直接识别并游玩。

## 分支建议
- track/xinput-stabilize：输入稳定性（去抖/滤波/日志降噪）
- track/input-latency：输入模式切换与上报节奏
- track/feature-extended-inputs：SL/SR、后续 IMU 与扩展映射
- track/refactor-input-pipeline：分层重构，减少重复解析
- track/research-native-driver：原生驱动可行性研究

## 建议执行顺序
1. 先稳定（xinput-stabilize）
2. 再降延迟（input-latency）
3. 再补功能（feature-extended-inputs）
4. 并行重构（refactor-input-pipeline）
5. 最后评估是否进入驱动开发（research-native-driver）
